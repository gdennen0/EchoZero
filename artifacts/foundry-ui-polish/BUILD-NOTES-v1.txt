Foundry functional v1 build notes

Changed in this run:
- Foundry dataset contracts now carry explicit `taxonomy`, `label_policy`, and `manifest` structures for percussion one-shot work. The default ingest path stamps `percussion.one_shot` taxonomy metadata, allowed-label policy, stable content hashes, richer folder provenance, and deterministic manifest ordering.
- Dataset planning is now train-ready instead of demo-only. Split plans persist per-sample assignments, planner metadata, label distribution, content-hash groups, and a leakage report. Duplicate content hashes are grouped globally so the same audio content cannot land in multiple splits.
- Dataset curation keeps the manifest and split plan coherent by removing rejected samples from deterministic ordering, split assignments, and grouped-content metadata in the derived version.
- Train run creation now validates the practical baseline path for one-shot training:
  - spec schema and required data/training fields
  - datasetVersionId and sampleRate consistency with the persisted dataset version
  - taxonomy targeting `percussion.one_shot`
  - label-policy compatibility with the requested classification mode
  - planned train/validation assignments present before a run can be created
  - no recorded duplicate-content leakage across splits
  - sane numeric constraints for `epochs`, `batchSize`, `learningRate`, `maxLength`, `hopLength`, and `fmax`
- Eval reports now persist baseline-oriented structure for metrics output: dataset version linkage, evaluated split name, aggregate metrics, per-class metrics, baseline metadata, threshold policy, confusion payloads, and a summary block.
- Artifact manifests are now tightened for runtime use. Finalized artifacts carry `datasetVersionId`, `specHash`, taxonomy, label policy, and runtime consumer/backend/device metadata. Compatibility checks now verify those fields against the originating run and dataset version in addition to preprocessing, class ordering, and relative `.pth` weights paths.
- Foundry tests were expanded end to end for the realistic path: dataset ingest/planning/curation, duplicate-hash split hygiene, run-spec validation, eval report persistence, artifact compatibility success/failure, CLI flow, and app-level smoke coverage.

Verification:
- `.venv\\Scripts\\python -m pytest -q tests/foundry` -> 14 passed

Remaining non-v1 items:
- No real trainer execution backend, checkpoint contents, or exporter pipeline was added; Foundry is now ready to hand a validated dataset/run/artifact contract set to training, but it still does not perform actual model optimization.
- No schema-engine integration was added; the contract JSON files were updated and the services enforce the practical baseline rules directly.
- Out of scope for this pass: broader timeline/UI systems, non-percussion taxonomies, advanced augmentation policies, and richer evaluation visualizations.
- TODO: add a dedicated dataset audit/report command that can surface duplicate-content groups and sparse-class warnings before planning. The data is persisted now, but there is not yet a first-class operator-facing report surface.
