    Check Status

    autosave_tick acquires lock ✅
    transaction() acquires lock ✅
    commit() acquires lock ✅

    Note: DirtyTracker itself has no internal lock. _dirty and _dirty_entity_ids are accessed from both main thread and autosave thread. In practice this is safe because: - is_dirty() reads a bool (atomic in CPython due to GIL) - clear() sets bool + clears set (called under session lock in autosave_tick) - mark_dirty() sets bool + adds to set (called from main thread)

    The GIL prevents actual corruption, but it's not formally thread-safe. LOW risk — would only matter if we move off CPython.

    TEST COVERAGE¶

    What's well-tested:¶

    • All CRUD operations for all 6 repos (73 persistence tests)
    • Session lifecycle: create, open, save, close, reopen (61 session tests)
    • Transaction commit + rollback
    • WAL mode, row_factory
    • Dirty tracker: all 5 mutation events + BlockStateChanged exclusion
    • Pipeline registry: all operations + edge cases (36 tests)
    • FK cascade deletes: project→song→version→layer→take chain
    • Autosave: commits dirty, skips clean, stops correctly
    • Crash recovery: detect, recover, discard

    Gaps Found:¶

    HIGH: 1. No test for PipelineConfigRepository — CRUD tests exist in test_persistence.py for all other repos but pipeline_config is not tested there. It IS tested indirectly via test_pipelines.py but deserves its own CRUD section.

    1. No concurrent thread test — autosave + main thread writing simultaneously. The locks are there but never stress-tested. Would catch lock ordering issues.

    2. get_schema_version uses row[0] — should use row['value'] or accept that it runs before row_factory is set.

    MEDIUM: 4. Repo properties create new instances each access — session.songs creates a new SongRepository(self.db) every call. If repos ever gained state, this breaks silently. Consider caching.

    1. init_db calls conn.executescript() which auto-commits — this means the initial schema creation commits outside the session's control. Acceptable for initialization but should be documented.

    2. No test for migration system — the infrastructure exists (_MIGRATIONS dict, apply_migrations) but there are zero migrations and zero migration tests. When the first migration lands, tests need to cover: empty DB → current version, and previous version → current version.

    LOW: 7. Private function imports — _serialize_take_data/_deserialize_take_data should be public.

    1. ProjectSession.open() stub — currently just delegates to open_db() with path hashing. Will need real .ez archive support in Phase 3.
